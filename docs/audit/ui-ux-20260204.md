## 1) Executive verdict

### Professional feel: **partial**
- Observed: UI has the right core screens and affordances (Onboarding, Side Panel, Summary, History, Settings, Diagnostics).  
  Evidence: `macapp/MeetingListenerApp/Sources/MeetingListenerApp.swift`, `macapp/MeetingListenerApp/Sources/SummaryView.swift`, `macapp/MeetingListenerApp/Sources/SessionHistoryView.swift`
- Observed: Side panel is dense but coherent (transcript/cards/entities + status and controls).  
  Evidence: `macapp/MeetingListenerApp/Sources/SidePanelView.swift`
- Inferred: The “stop → summary” experience will feel professional once failure outcomes are explicit and predictable.

### Trust/privacy: **partial**
- Observed: Permissions are surfaced and actionable (Open Settings deep links, banners).  
  Evidence: `macapp/MeetingListenerApp/Sources/OnboardingView.swift`, `macapp/MeetingListenerApp/Sources/SidePanelView.swift`
- Observed: Visible state exists (Listening/Idle, Streaming/Reconnecting, audio meters).  
  Evidence: `macapp/MeetingListenerApp/Sources/SidePanelView.swift`, `macapp/MeetingListenerApp/Sources/MeetingListenerApp.swift`
- Inferred: Trust still hinges on “What exactly did you capture?” and “Did I lose anything when stopping?” being unambiguous.

### Biggest UX adoption risk (P0)
- **Finalization ambiguity when backend doesn’t send a final summary** (timeout, backend error, disconnect). Users won’t know if notes are complete vs partial.  
  Evidence: `macapp/MeetingListenerApp/Sources/AppState.swift` (stop flow), `macapp/MeetingListenerApp/Sources/WebSocketStreamer.swift` (timeout path)

### Biggest polish opportunity (P1)
- **Entity UX**: highlight modes are powerful but need explanation; entity lane should be clickable to filter/jump so users discover the feature.  
  Evidence: `macapp/MeetingListenerApp/Sources/SidePanelView.swift`, `macapp/MeetingListenerApp/Sources/EntityHighlighter.swift`

## 4) Screen-by-screen critique

### Screen: Onboarding (Window: "Welcome to EchoPanel")
**Purpose + primary action**
- Observed: Explain permissions + configure sources + optional diarization token + start listening.  
  Evidence: `macapp/MeetingListenerApp/Sources/OnboardingView.swift`

**What works**
- Observed: Clear step indicator, single CTA, deep-links to System Settings.  
  Evidence: `macapp/MeetingListenerApp/Sources/OnboardingView.swift`

**What breaks**
- Inferred: The permissions step does not adapt messaging to the chosen audio source (System vs Mic vs Both). This can create confusion about “required vs optional”.  
  Evidence: `macapp/MeetingListenerApp/Sources/OnboardingView.swift` (permission rows are static), `macapp/MeetingListenerApp/Sources/AppState.swift` (source-specific permission logic)

**Scores**
- Productivity: 7/10 (Observed: fast flow; Inferred: better source-aware copy improves)
- Trust: 7/10 (Observed: deep-links; Inferred: clarify “why” by source)
- Polish: 6/10 (Observed: functional; Inferred: more tailored states)

**Top recommendations**
1) Source-aware permission framing (P1)
   - What: If source == microphone-only, mark Screen Recording “not needed”; if system-only, mark Microphone “not needed”.
   - Where: `OnboardingView.permissionsStep`
   - Why: reduces false “permission anxiety”
   - Validate: change source and see correct required badges
