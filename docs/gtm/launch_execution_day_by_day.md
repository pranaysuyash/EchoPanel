### 3.3 Launch Day FAQ Preparation

**Common Questions to Prepare For:**

**Q1: "Is this really local-only? How do you prove it?"**
```
A: Yes, truly local-only. You can verify by:

1. Open Activity Monitor during a session
2. Monitor network activity
3. You'll see ZERO network calls to EchoPanel domains
4. All processing happens via MLX (Metal-accelerated) on your machine

Source code available for audit. No cloud backend required for transcription.
```

**Q2: "How does accuracy compare to Whisper/Otter?"**
```
A: In our testing, MLX Qwen3-ASR achieves:
- 90%+ accuracy on clear audio
- Comparable to Whisper base
- Better performance on Apple Silicon (Metal acceleration)

We support multiple models (MLX, Whisper) so you can choose what works best for your use case.
```

**Q3: "Why isn't this open source?"**
```
A: Valid question. As a solo developer building sustainable business, I'm keeping source closed for now to fund development.

However:
- All architecture is documented
- I'm very transparent about how it works
- Long-term, I may open source components
```

**Q4: "Will this stay free or become paid?"**
```
A: EchoPanel has a free tier (5 meetings/month) for those who want to try it.

Pro tier ($12/month or $79 lifetime) offers:
- Unlimited sessions
- All ASR models
- All export formats
- Priority support

Free tier remains free forever for light users.
```

**Q5: "What happens if you shut down the service?"**
```
A: Your data remains on your device. Because EchoPanel runs locally:

- You retain all your transcripts
- You can export all data anytime
- No cloud dependency means your data is not at risk

Lifetime purchases (if App Store allows) would continue working as long as you have the app installed.
```

## 6. Crisis Response Playbook

### 6.1 Potential Crisis Scenarios

**Crisis 1: App Store Rejection**

**Scenario:** App Store rejects EchoPanel submission

**Impact:** Cannot launch on App Store, major revenue channel blocked

**Response Plan:**
1. **Understand Rejection:**
   - Read rejection reasons carefully
   - Document specific guidelines violated
   - Identify if fixable or fundamental issue

2. **Immediate Fixes:**
   - Fix all identified issues (24-48 hours)
   - Resubmit quickly
   - Communicate delay to community

3. **Alternative Launch:**
   - Launch via direct download (.dmg from website)
   - Offer beta via TestFlight
   - Proceed with Product Hunt launch (independent of App Store)

4. **Communication:**
   - Twitter: "App Store reviewing, launching via direct download in meantime"
   - Email waitlist: Provide direct download link
   - Update landing page: Direct download CTA

**Crisis 2: Major Bug on Launch Day**

**Scenario:** Critical bug discovered (app crashes, doesn't work, data loss)

**Impact:** Users frustrated, negative reviews, brand damage

**Response Plan:**
1. **Immediate Assessment:**
   - Reproduce bug
   - Identify severity (crash vs. minor issue)
   - Determine if hotfix needed or can wait

2. **Communication:**
   - Twitter: "We're investigating a critical issue. We'll have a fix ASAP."
   - Product Hunt: Comment explaining issue and ETA
   - Email: Acknowledge issue to all new signups

3. **Hotfix Process:**
   - Prioritize fix above all other work
   - Release hotfix within 24-48 hours
   - Test thoroughly before release
   - Document what happened

4. **Follow-Up:**
   - Post-mortem blog post: "What Happened on Launch Day"
   - Compensate affected users (free month, lifetime upgrade)
   - Update App Store with fix

**Crisis 3: Negative Viral Feedback**

**Scenario:** EchoPanel gets negative viral attention (privacy concerns, bug reports, criticism)

**Impact:** Brand damage, lost trust, reduced conversions

**Response Plan:**
1. **Monitor:**
   - Track mentions across all channels
   - Identify primary complaints
   - Categorize: Valid concerns vs. misunderstandings vs. trolls

2. **Address Valid Concerns:**
   - Acknowledge: "You're right, we made a mistake"
   - Explain: "Here's what happened and why"
   - Fix: Implement fix ASAP
   - Update: Communicate fix

3. **Clarify Misunderstandings:**
   - Be patient: "Thanks for bringing this up, here's clarification"
   - Provide evidence: Screenshots, docs, code
   - Be transparent: "This is what we do and don't do"

4. **Ignore Trolls:**
   - Don't engage (no fuel for them)
   - Focus on constructive feedback
   - Moderate comments if needed

**Crisis 4: Competitive Attack**

**Scenario:** Competitor spreads misinformation or attacks EchoPanel

**Impact:** Brand damage, lost trust

**Response Plan:**
1. **Assess:**
   - Verify claims are false
   - Document evidence
   - Identify if legitimate concern or attack

2. **If Legitimate Concern:**
   - Address honestly (see Crisis 3)
   - Fix issue if needed

3. **If Attack/Misinformation:**
   - Correct facts calmly: "For clarity, EchoPanel does [clarification]"
   - Provide evidence: Source code, docs, third-party validation
   - Don't attack back (maintain high road)

4. **Monitor:**
   - Track spread of misinformation
   - Repeat correction as needed
   - Engage community to defend (if they choose to)

**Crisis 5: Server/Backend Outage**

**Scenario:** Backend service goes down (if any cloud component)

**Impact:** Users can't use app, frustration

**Response Plan:**
1. **Immediate Communication:**
   - Twitter: "We're experiencing issues with [service]. Working on it."
   - Email: Update to recent signups
   - Landing page: Status banner

2. **Investigation & Fix:**
   - Identify root cause
   - Implement fix ASAP
   - Test thoroughly

3. **Post-Incident:**
   - Update: "Issue resolved. Details here: [link to post-mortem]"
   - Compensation: Offer free month to affected users
   - Post-mortem: Document what happened and how to prevent

### 6.2 Crisis Communication Guidelines

**General Principles:**
- **Speed:** Respond within 1 hour for major issues
- **Transparency:** Be honest about what happened
- **Empathy:** Acknowledge user frustration
- **Action-Oriented:** Focus on fixes and next steps
- **Updates:** Provide regular status updates

**Tone:**
- Professional (not defensive)
- Apologetic when appropriate (not over-apologetic)
- Confident in ability to fix
- Grateful for feedback (even criticism)
